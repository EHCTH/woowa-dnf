## 주요 기능

### 구현 목적
- 현재 던파 세팅/데미지 계산에 가장 많이 사용되는 사이트는 **던담**이라고 생각합니다. 던담의 데미지 계산 로직이 사실상 보편적인 기준처럼 쓰이고 있습니다.
- 다만 직업마다 성능 차이가 크고, 같은 세팅이라도 캐릭터마다 체감이 달라 **하나의 데미지 로직만으로 모든 캐릭터를 평가하기에는 한계가 있다**고 느꼈습니다.(공식 자체가 틀렸다고 보는 것은 아닙니다.)
- 그래서 추후에 개발 예정인 내용은 다음과 같습니다.
    - 아이템/세팅 정보만을 사용하는 **커스텀 데미지 공식**
    - 던담 방식과 유사한 **기존 데미지 공식**
- 두 가지를 나란히 지원하는 것을 목표로 하고 있으며, 현재는 **조회 중심 기능**을 먼저 구현해 둔 상태입니다.

### 캐릭터 검색
- 서버 + 캐릭터명으로 단일 캐릭터 검색
- 모험단 이름으로 전체 캐릭터 검색
- 길드 이름으로 전체 캐릭터 검색

### 캐릭터 상세 조회
- **기본 정보**: 서버, 레벨, 직업, 명성, 모험단명, 길드명 등
- **장비 정보**: 무기 / 방어구 / 악세서리 / 보조장비 / 휘장
- **버프 장비**: 버프용 장비 세트, 버프 스킬 정보
- **아바타 & 버프 아바타**: 부위별 아바타, 엠블렘 정보
- **크리쳐 & 버프 크리쳐**: 크리쳐, 아티팩트, 스킬 정보
- **휘장(Flag)**: 휘장 옵션 정보

### 병렬 조회 & 스냅샷
- 초기 구현에서는 외부 API를 순차적으로 호출하여, **캐릭터 1명 조회에 약 2~3초**가 소요되었습니다.
- 이를 개선하기 위해 Java 21 Virtual Thread + `CompletableFuture` 로  
  장비, 아바타, 버프, 크리쳐 등 여러 정보를 **병렬 + 비동기 처리**하도록 변경했습니다.
  - 캐릭터 정보를 **스냅샷 형태로 저장**하여 이후 조회 시 재사용할 수 있도록 설계했습니다.

### 동작 방식
외부 API와 DB를 조합해서 캐릭터 정보를 다루는 흐름은 다음과 같습니다.
```
[저장]
외부 API(JSON)
  -> External DTO
  -> DOMAIN(캐릭터)
  -> SNAPSHOT(저장용 엔티티)
  -> 저장(MySQL / 메모리)

[조회]
조회(MySQL / 메모리)
  -> SNAPSHOT(저장용 엔티티)
  -> DOMAIN(캐릭터)
  -> View DTO(화면용)
```

### 채팅 기능
- 캐릭터 상세 페이지 우측 탭에 **실시간 채팅**을 제공합니다.
- 캐릭터별 방(서버 ID + 캐릭터 ID)로 분리된 채팅방 구조를 사용합니다.
